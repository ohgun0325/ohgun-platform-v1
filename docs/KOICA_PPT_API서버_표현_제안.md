# PPT 통합 시스템 흐름도 – API 서버 표현 개선 제안

현재 슬라이드의 **API 서버** 블록이 "요청 분배 (Request Distribution)"로만 되어 있어, **무엇을 어떻게 분배하는지**가 불명확합니다.  
실제 구현에 맞추어 표현을 바꾸는 방안을 제안합니다.

---

## 1. 왜 "요청 분배"가 애매한가

- **요청 분배**라고 하면 청중이 떠올리기 쉬운 것**
  - 로드 밸런싱, 작업 큐(Job Queue), 여러 워커로 나누어 보내기 등
- **현재 KOICA 구현**
  - Job Queue 없음, 요청당 **동기 처리**
  - 클라이언트가 **엔드포인트를 선택**해서 호출 (`/detect`, `/ocr`, `/excel` 등)
  - API 서버는 **해당 경로에 연결된 처리기(검출/OCR/Excel)로 요청을 넘기는 역할**만 수행

그래서 "분배"보다는 **"수신 → 처리 유형별 전달"**에 가깝게 쓰는 편이 정확하고 이해하기 쉽습니다.

---

## 2. API 서버 블록 문구 제안

### 옵션 A – 역할 중심 (추천)

**메인 문구**
- **요청 수신 및 처리 유형별 전달**

**부가 설명(슬라이드 또는 발표 시)**
- "사용자가 업로드한 파일을 받아, 선택한 기능(인감·서명 검출 / OCR / Excel 추출)에 맞는 AI 엔진으로 전달합니다."

- **영문** (필요 시): Request reception & routing by processing type

---

### 옵션 B – 비기술자 친화

**메인 문구**
- **파일 수신 및 AI 처리 연결**

**부가 설명**
- "업로드된 파일을 받아, 해당하는 AI 기능(검출·OCR·Excel)으로 넘기는 구간입니다."

---

### 옵션 C – 엔드포인트 강조 (기술 문서용)

**메인 문구**
- **API 엔드포인트 제공 및 처리기 연결**

**부가 설명**
- "각 기능별 API(/detect, /ocr, /excel)로 들어온 요청을 해당 AI 엔진과 연결합니다."

---

### 옵션 D – 한 줄만 바꿀 때 (최소 수정)

**기존:** 요청 분배 (Request Distribution)  
**변경:** **요청 수신 및 처리기 연결** (Receive & route to handler)

---

## 3. 슬라이드에 넣기 좋은 한 줄 정리

| 구분 | 문구 |
|------|------|
| **한글 메인** | 요청 수신 및 처리 유형별 전달 |
| **한글 서브** | 파일 업로드 → 검출/OCR/Excel 중 해당 AI 엔진으로 전달 |
| **영문** | Request reception & routing by processing type |

---

## 4. 정리

- **"요청 분배"** 는 로드밸런싱·작업큐를 연상시켜 현재 구조와 맞지 않습니다.
- **"요청 수신 및 처리 유형별 전달"** 또는 **"파일 수신 및 AI 처리 연결"** 로 바꾸면,
  - 실제 동작(경로별 라우팅, 동기 처리)과 맞고
  - 청중이 "파일 받아서 어떤 처리로 넘기는지"를 직관적으로 이해하기 쉽습니다.

발표 대상(경영진/비개발 vs 개발팀)에 따라 **옵션 A** 또는 **옵션 B**를 메인으로 쓰고, 나머지는 슬라이드 노트나 부가 설명으로 두면 됩니다.
